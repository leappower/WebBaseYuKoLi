## 6.8 新增：并行开发策略

在重构期间（~22d），业务开发采用 **feature branch 隔离**策略：
1. 所有重构在 `dev-refactor-*` 分支上进行
2. 业务新功能在 `dev-*` 分支上开发
3. 每周合并一次 `dev-refactor` → `dev`（如果有冲突说明业务代码耦合了重构目标）
4. 如果业务需要快速上线 → 回滚重构分支，使用旧的 IIFE 构建方式

## 8. 风险与备选方案

### 风险矩阵

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 废弃 spa-router 导致未发现的依赖崩溃 | 中 | 高 | T0.2 逐步移除，非一次性删除 |
| 团队抵抗 ESM 迁移 | **高** | 中 | 迁移指南 + pair programming + 过渡双出口 |
| 阶段零+一代码重叠导致合并冲突 | **高** | 中 | 明确同一人负责（GLM 问题 17）|
| SWUP 事件迁移导致回归 | 中 | 高 | 分两步：先兼容，后统一 |
| 翻译 JSON 网络延迟 > when-ready 超时 | 低 | 中 | preload link + 合理的超时值(5s) |
| 重构期间并行业务开发阻塞 | 中 | 高 | feature branch 隔离（§6.8） |
| 翻译 JSON 文件过大 >50KB | 中 | 中 | 压缩 + 分语言按需加载 |

### 备选方案

| 场景 | 方案 |
|------|------|
| 阶段零后发现 SWUP 不适合 → 回退到 spa-router | **修正**: 需要 5-7 天而非 3 天（GLM 问题 18，原备选方案过于乐观）|
| 阶段二发现自建 i18n 无法模块化 | 直接用 i18next 替换（+2d） |
| 阶段三 ESM 迁移引起兼容性问题 | 保留 IIFE 入口 + ESM 双出口 |
| 老板觉得 22 天太长 | 简化版 6 天：阶段零+一+二，阶段三+四+五延后 |

### "不做"的代价

如果什么都不做，预期 60 天内：
1. `window.t()` 时序 bug **翻倍**（从 ~10 到 ~20+）
2. 每次新页面类型加入需要 **5+ 轮时序调试**
3. SWUP 升级 → `__swupEnabled` 行为改变 → 双路由器共存逻辑崩溃
4. **新的时序补丁继续积累**（GLM 问题 15），62 DOMContentLoaded → ~90，108 setTimeout → ~150
5. LeFthook / lint 无法阻止架构级问题
